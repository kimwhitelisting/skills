# MCP 서버 모범 사례

## 빠른 참조

### 서버 이름 지정
- **파이썬**: `{service}_mcp`(예: `slack_mcp`)
- **노드/타입스크립트**: `{service}-mcp-server`(예: `slack-mcp-server`)

### 도구 이름 지정
- 서비스 접두사와 함께 snake_case 사용
- 형식 : `{service}_{action}_{resource}`
- 예: `slack_send_message`, `github_create_issue`

### 응답 형식
- JSON 및 Markdown 형식을 모두 지원합니다.
- 프로그래밍 처리를 위한 JSON
- 사람의 가독성을 위한 마크다운

### 페이지 매김
- 항상 `limit` 매개변수를 존중합니다.
- `has_more`, `next_offset`, `total_count` 반환
- 기본 20~50개 항목

### 교통
- **스트리밍 가능한 HTTP**: 원격 서버의 경우 다중 클라이언트 시나리오
- **stdio**: 로컬 통합의 경우 명령줄 도구
- SSE 방지(스트리밍 가능한 HTTP를 위해 더 이상 사용되지 않음)

## 서버 명명 규칙

다음과 같은 표준화된 명명 패턴을 따르십시오.

**Python**: `{service}_mcp` 형식 사용(밑줄이 있는 소문자)
- 예: `slack_mcp`, `github_mcp`, `jira_mcp`

**Node/TypeScript**: `{service}-mcp-server` 형식 사용(하이픈 포함 소문자)
- 예: `slack-mcp-server`, `github-mcp-server`, `jira-mcp-server`

이름은 일반적이어야 하며, 통합되는 서비스를 설명하고, 작업 설명에서 쉽게 추론할 수 있어야 하며, 버전 번호가 없어야 합니다.

## 도구 이름 지정 및 디자인

### 도구 이름 지정

1. **snake_case 사용**: `search_users`, `create_project`, `get_channel_info`
2. **서비스 접두사 포함**: MCP 서버가 다른 MCP 서버와 함께 사용될 수 있다고 예상합니다.
- `send_message` 대신 `slack_send_message`을 사용하세요.
- `create_issue` 대신 `github_create_issue`을 사용하세요.
3. **행동 지향적이어야 합니다**: 동사(get, list, search, create 등)로 시작하세요.
4. **구체적이어야 합니다**: 다른 서버와 충돌할 수 있는 일반적인 이름은 피하세요.

### 도구 디자인

- 도구 설명은 기능을 좁고 명확하게 설명해야 합니다.
- 설명은 실제 기능과 정확히 일치해야 합니다.
- 도구 주석 제공(readOnlyHint, destructiveHint, idempotentHint, openWorldHint)
- 도구 작업을 집중적이고 원자적으로 유지합니다.

## 응답 형식

데이터를 반환하는 모든 도구는 다양한 형식을 지원해야 합니다.

### JSON 형식(`response_format="json"`)
- 기계가 읽을 수 있는 구조화된 데이터
- 사용 가능한 모든 필드와 메타데이터를 포함합니다.
- 일관된 필드 이름 및 유형
- 프로그래밍 처리에 사용

### 마크다운 형식(`response_format="markdown"`, 일반적으로 기본값)
- 사람이 읽을 수 있는 형식의 텍스트
- 명확성을 위해 헤더, 목록, 서식을 사용하세요.
- 타임스탬프를 사람이 읽을 수 있는 형식으로 변환
- 괄호 안에 ID가 포함된 표시 이름 표시
- 장황한 메타데이터 생략

## 페이지 매김

리소스를 나열하는 도구의 경우:

- **항상 `limit` 매개변수를 존중하세요**
- **페이지 매김 구현**: `offset` 또는 커서 기반 페이지 매김을 사용합니다.
- **페이지 매기기 메타데이터 반환**: `has_more`, `next_offset`/`next_cursor`, `total_count` 포함
- **모든 결과를 메모리에 로드하지 마세요**: 대규모 데이터 세트의 경우 특히 중요합니다.
- **합리적인 제한의 기본값**: 일반적으로 20~50개 항목

페이지 매기기 응답 예시:
```json
{
  "total": 150,
  "count": 20,
  "offset": 0,
  "items": [...],
  "has_more": true,
  "next_offset": 20
}
```

## 운송 옵션

### 스트리밍 가능한 HTTP

**최적의 용도**: 원격 서버, 웹 서비스, 다중 클라이언트 시나리오

**형질**:
- HTTP를 통한 양방향 통신
- 여러 동시 클라이언트 지원
- 웹 서비스로 배포 가능
- 서버-클라이언트 알림 활성화

**다음 경우에 사용**:
- 동시에 여러 클라이언트에 서비스 제공
- 클라우드 서비스로 배포
- 웹 애플리케이션과의 통합

### 스튜디오

**최적의 용도**: 로컬 통합, 명령줄 도구

**형질**:
- 표준 입출력 스트림 통신
- 간단한 설정, 네트워크 구성 불필요
- 클라이언트의 하위 프로세스로 실행됩니다.

**다음 경우에 사용**:
- 로컬 개발 환경을 위한 도구 구축
- 데스크탑 애플리케이션과 통합
- 단일 사용자, 단일 세션 시나리오

**참고**: stdio 서버는 stdout에 기록하면 안 됩니다(로깅에는 stderr 사용).

### 운송 선택

| 기준 | 스튜디오 | 스트리밍 가능한 HTTP |
|-----------|-------|-----------------|
| **배포** | 지역 | 원격 |
| **클라이언트** | 싱글 | 다중 |
| **복잡성** | 낮음 | 중간 |
| **실시간** | 아니요 | 예 |

## 보안 모범 사례

### 인증 및 승인

**OAuth 2.1**:
- 공인 기관의 인증서와 함께 안전한 OAuth 2.1을 사용하세요.
- 요청을 처리하기 전에 액세스 토큰의 유효성을 검사합니다.
- 귀하의 서버를 위해 특별히 고안된 토큰만 허용합니다.

**API 키**:
- API 키를 코드가 아닌 환경 변수에 저장합니다.
- 서버 시작 시 키 유효성 검사
- 인증 실패 시 명확한 오류 메시지 제공

### 입력 유효성 검사

- 디렉터리 탐색을 방지하기 위해 파일 경로를 삭제합니다.
- URL 및 외부 식별자의 유효성을 검사합니다.
- 매개변수 크기 및 범위 확인
- 시스템 호출 시 명령 주입 방지
- 모든 입력에 대해 스키마 유효성 검사(Pydantic/Zod)를 사용합니다.

### 오류 처리

- 내부 오류를 클라이언트에게 노출하지 마세요.
- 서버측에서 보안 관련 오류를 기록합니다.
- 도움이 되지만 드러나지 않는 오류 메시지를 제공하세요.
- 오류 발생 후 리소스 정리

### DNS 리바인딩 보호

로컬로 실행되는 스트리밍 가능한 HTTP 서버의 경우:
- DNS 리바인딩 보호 활성화
- 들어오는 모든 연결에서 `Origin` 헤더의 유효성을 검사합니다.
- `0.0.0.0` 대신 `127.0.0.1`에 바인딩

## 도구 주석

고객이 도구 동작을 이해하는 데 도움이 되도록 주석을 제공합니다.

| 주석 | 유형 | 기본값 | 설명 |
|-----------|------|---------|-------------|
| `readOnlyHint` | 부울 | 거짓 | 도구는 환경을 수정하지 않습니다 |
| `destructiveHint` | 부울 | 사실 | 도구가 파괴적인 업데이트를 수행할 수 있음 |
| `idempotentHint` | 부울 | 거짓 | 동일한 인수를 사용한 반복 호출에는 추가 효과가 없습니다 |
| `openWorldHint` | 부울 | 사실 | 도구는 외부 엔터티와 상호 작용 |

**중요**: 주석은 보안 보장이 아니라 힌트일 뿐입니다. 클라이언트는 주석에만 의존하여 보안에 중요한 결정을 내려서는 안 됩니다.

## 오류 처리

- 표준 JSON-RPC 오류 코드 사용
- 결과 개체 내의 도구 오류 보고(프로토콜 수준 오류 아님)
- 제안된 다음 단계와 함께 유용하고 구체적인 오류 메시지 제공
- 내부 구현 세부정보를 노출하지 마세요.
- 오류 발생 시 리소스를 적절하게 정리합니다.

오류 처리 예시:
```typescript
try {
  const result = performOperation();
  return { content: [{ type: "text", text: result }] };
} catch (error) {
  return {
    isError: true,
    content: [{
      type: "text",
      text: `Error: ${error.message}. Try using filter='active_only' to reduce results.`
    }]
  };
}
```

## 테스트 요구 사항

포괄적인 테스트에는 다음이 포함되어야 합니다.

- **기능 테스트**: 유효한/잘못된 입력으로 올바른 실행을 확인합니다.
- **통합 테스트**: 외부 시스템과의 상호 작용 테스트
- **보안 테스트**: 인증 유효성 검사, 입력 삭제, 속도 제한
- **성능 테스트**: 로드 시 동작, 시간 초과 확인
- **오류 처리**: 적절한 오류 보고 및 정리를 보장합니다.
